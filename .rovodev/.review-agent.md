# Code Review Guidelines for BILDIT Web Next.js

## General Principles

- Prioritize code clarity and maintainability
- Follow the DRY (Don't Repeat Yourself) principle
- Write self-documenting code with meaningful names
- Keep functions small and focused on a single responsibility
- Consider future maintainers when making decisions

## TypeScript Standards

- **Type Safety**: Ensure all functions, components, and variables have proper type annotations
- **Strict Mode**: The project uses TypeScript strict mode - ensure all code complies
- **Avoid `any`**: Use `unknown` or proper types instead of `any`. If `any` is necessary, provide justification
- **Type Imports**: Use `import type` for type-only imports to improve build performance
- **Interface vs Type**: Prefer `interface` for object shapes, use `type` for unions, intersections, and computed types

## Next.js Best Practices

- **Server Components**: Default to Server Components unless client-side interactivity is required
- **Client Components**: Use `'use client'` directive only when necessary (hooks, event handlers, browser APIs)
- **File Conventions**: Follow Next.js App Router conventions:
  - `page.tsx` for routes
  - `layout.tsx` for layouts
  - `loading.tsx` for loading states
  - `error.tsx` for error boundaries
  - `not-found.tsx` for 404 pages
- **API Routes**: Ensure proper error handling and status codes in API routes
- **Metadata**: Include appropriate metadata for SEO when creating new pages
- **Image Optimization**: Use Next.js `Image` component for images when possible

## React Patterns

- **Component Structure**: 
  - Use functional components with hooks
  - Keep components focused and composable
  - Extract reusable logic into custom hooks
- **State Management**: 
  - Prefer local state (`useState`) for component-specific state
  - Use context sparingly for truly global state
  - Consider server state management patterns for data fetching
- **Performance**:
  - Use `React.memo` for expensive components that re-render frequently
  - Implement proper dependency arrays in `useEffect`, `useMemo`, and `useCallback`
  - Avoid unnecessary re-renders
- **Hooks Rules**: Follow React Hooks rules (only call hooks at the top level)

## Code Style & Formatting

- **Prettier**: All code must be formatted with Prettier (configured in ESLint)
- **Formatting Rules**:
  - Single quotes for strings
  - No semicolons
  - Print width: 120 characters
  - Tab width: 2 spaces
  - Trailing commas: none
  - Arrow function parentheses: always
- **Naming Conventions**:
  - Components: PascalCase (e.g., `HeroImage.tsx`)
  - Functions/Variables: camelCase (e.g., `getPostData`)
  - Constants: UPPER_SNAKE_CASE (e.g., `API_BASE_URL`)
  - Files: Match the export (PascalCase for components, camelCase for utilities)
  - **Unauthenticated/Public Functions**: Functions that are unauthenticated or public should include "remote" in their name (e.g., `remoteGetData`, `fetchRemotePosts`)
  - **Scheduled Functions**: Scheduled functions should be named `scheduledFunctions` (e.g., `scheduledFunctions`, `scheduledCleanupTasks`)
  - **Utility Functions**: Utility functions should have a "utility" prefix (e.g., `utilityFormatDate`, `utilityParseString`)
- **Import Order**: Follow the configured import order:
  1. React
  2. Ant Design (if used)
  3. Styled components
  4. Third-party modules
  5. Local imports (starting with `/`)

## Security

- **Input Validation**: Validate and sanitize all user inputs
- **API Security**: Never expose sensitive data in client-side code
- **Environment Variables**: Use Next.js environment variables for secrets (never commit `.env.local`)
- **XSS Prevention**: Use React's built-in XSS protection; avoid `dangerouslySetInnerHTML` unless absolutely necessary
- **CSRF Protection**: Ensure API routes have proper CSRF protection when needed

## Performance

- **Code Splitting**: Leverage Next.js automatic code splitting
- **Lazy Loading**: Use dynamic imports for heavy components (`next/dynamic`)
- **Image Optimization**: Always use optimized images (Next.js Image component or proper formats)
- **Bundle Size**: Be mindful of bundle size; avoid unnecessary large dependencies
- **Server Components**: Prefer Server Components to reduce client-side JavaScript

## Error Handling

- **Error Boundaries**: Implement error boundaries for graceful error handling
- **API Errors**: Handle API errors appropriately with user-friendly messages
- **Type Errors**: Ensure TypeScript errors are resolved, not suppressed
- **Logging**: Log errors appropriately for debugging (consider using proper logging service)

## Testing Considerations

- **Test Coverage**: Ensure new features have appropriate test coverage
- **Unit Tests**: Write unit tests for utility functions and complex logic
- **Integration Tests**: Consider integration tests for critical user flows
- **Accessibility**: Ensure components are accessible (proper ARIA labels, keyboard navigation)

## Documentation

- **Code Comments**: Add comments explaining "why" not "what" - code should be self-explanatory
- **No Code Blocks in Comments**: Do not use code blocks (triple backticks) within comments - use inline code with single backticks or plain text instead
- **Complex Logic**: Document complex algorithms or business logic
- **Component Props**: Use TypeScript interfaces/types to document component props
- **API Documentation**: Document API routes with clear parameter and response types

## Common Issues to Flag

- **Dead Code**: Remove unused imports, variables, and functions
- **Console Statements**: Remove `console.log` statements (use proper logging in production)
- **Commented Code**: Remove commented-out code (use git history instead)
- **Code Blocks in Comments**: Flag any code blocks (triple backticks) used within comments - use inline code with single backticks instead
- **Magic Numbers**: Replace magic numbers with named constants
- **Hardcoded Values**: Avoid hardcoded strings/values that should be configurable
- **Inconsistent Patterns**: Ensure consistency with existing codebase patterns
- **Missing Error Handling**: Ensure all async operations have error handling
- **Type Assertions**: Avoid `as` type assertions unless absolutely necessary; prefer type guards

## BILDIT-Specific Considerations

- **Banner Components**: Follow established patterns for banner/hero image components
- **Provider Patterns**: Maintain consistency with existing provider patterns
- **Middleware**: Ensure middleware changes don't break existing functionality

## Review Checklist

When reviewing code, verify:
- [ ] Code follows TypeScript strict mode requirements
- [ ] All imports are properly organized
- [ ] Code is formatted with Prettier
- [ ] No console.log statements remain
- [ ] Error handling is implemented
- [ ] Components follow Next.js App Router conventions
- [ ] Server/Client component boundaries are correct
- [ ] No security vulnerabilities introduced
- [ ] Performance considerations are addressed
- [ ] Code is maintainable and follows DRY principles
- [ ] Documentation is adequate for complex logic
